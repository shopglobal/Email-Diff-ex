Specification:
This here Shiz is a function that removes all duplicates from an unsorted list of email addresses. The constraints of this shiz are:

  • Leave the resulting list in the original order (same as the input).
  • Be able to handle 100,000 email addresses containing 50% randomly placed duplicates in well under 1 second on a typical modern laptop.

Precondition:
The only pre-condition I've placed on the input list is that each item stands alone on a line, so just one email per line please. (Technically speaking, each line should be of the form: [emal][\n] meaning an email address followed immediately by the newline character).

Gratuitous, Self-Serving Analysis:
This is acceptably good shiz. Tested with a list of 200,000 randomly generated "emails" where every odd item was a duplicate. So, that's a list of 200,000 with 100,000 duplicates (50%). Even for a list of 200,000, this here shiz was finished in under 500ms total runtime. It was also tested against a fairly smooth curve of approximately 100,100 total items with a dispersal of approx. 216 original emails, each occuring up to 500 times each. The algorythm took approx 112.184ms, with an overall process runtime of 232ms. This isn't the best shiz, as the algorythm used was naieve, but good enough for the constraints of the problem.


The Algorithm, annotated...

The algorithm was turned into a node.js module for obvious reasons. Possibly at first blush, an unbovious require is that of the bluebird promise library. Upon reflection though, it will seem quite obvious: When we want to send the file to the browser, reading it and piping it as it's read is a cecidedly asynchronous operation. The algorithm's module "promisified" the functions so that they would be "then-able", and so allow for this.

To protect against a memory crash in the case of crazy huge files and server memory load, the file is read in as a stream, line by line.

As each line is read in, the line (email address) is made the key of a hash.
If we haven't seen this key before, it is assigned the value of 1. A clever increment by one (i.e., hash[line]++) can't be used for the initial encounter due to it being undefined.
If we have seen it before, its key is incremented.
That's it... That's the crux of the algorithm. Underwhelming, I know.

                return new Promise(function(resolve) {   
                  console.time('foo');
                  readByLine.on('line', function(line){
                    if( typeof hash[line] == "undefined" ) {
                      hash[line] = 1;
                    } else {
                      hash[line]++;
                    }
                  });

The workings of this function are encased in a Bluebird Promise When all the lines have been read, a close event is fired.
Since the file is closed, the function can resolve the Promise. 

                      readByLine.on('close', function(){

                      });

Now we run back through the hash.
If there is more than one occurence of the current line (email), we only write one to the file.
Again, this is done as a stream, and so is "promisified". Doing it this way, line by line, satisfies the condition of
writing it back out in the original order.
Not rocket science. First, make it work. Then, if needed, make it fast...
